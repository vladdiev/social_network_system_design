# Social Network for Travelers — System Design

System design of a social network for travelers, created for the Balun System Design course (https://balun.courses/courses/system_design).

## Функциональные требования

### Профиль
- Пользователь может просматривать профиль другого пользователя
- Пользователь может редактировать свой профиль (информация о себе, фото)
- Пользователь может удалить свой профиль (контент сохраняется анонимизированным как "Удаленный пользователь")

### Посты
- Пользователь может создать пост (фото, описание, привязка к месту)
- Пользователь может просмотреть пост
- Пользователь может отредактировать свой пост
- Пользователь может удалить свой пост

### Комментарии
- Пользователь может создать, просмотреть, отредактировать и удалить комментарий под постом

### Оценки
- Пользователь может поставить и убрать оценку (лайк/дизлайк) на посты других пользователей

### Подписки
- Пользователь может подписаться на другого пользователя и отписаться

### Лента
- Пользователь может просматривать свою ленту по подпискам в обратном хронологическом порядке
- Пользователь может просматривать ленту другого пользователя

### Места
- Пользователь может искать популярные места для путешествий
- Пользователь может просматривать посты конкретного места

## Нефункциональные требования

- Аудитория
  - 10 000 000 DAU (целевая точка через год после запуска)
  - Рост линейный, ~+10 000 000 DAU в год
  - Горизонт планирования 2 года -> проектная емкость 20 000 000 DAU (запас x2); расчет RPS ведем на 10 000 000 DAU
- Регионы использования приложения
  - СНГ

- Поведение пользователей

Средние дневные действия среднего пользователя (read/write):

  - Профиль:
    - Просмотр профиля — 3 (read)
    - Редактирование/удаление профиля — редкие write, в оценке нагрузки пренебрегаем
  - Посты:
    - Публикация поста — 0.01 (write) (1 пост на 100 человек)
    - Редактирование/удаление поста — редкие write, пренебрегаем
  - Комментарии:
    - Создание комментария — 0.05 (write) (1 коммент на 20 человек)
    - Чтение комментариев — 5 (read)
    - Редактирование/удаление комментария — редкие write, пренебрегаем
  - Оценки:
    - Поставить/убрать оценку — 0.5 (write) (каждый второй ставит)
  - Подписки:
    - Подписка/отписка — 0.01 (write)
  - Лента:
    - Просмотр своей ленты — 5 (read)
    - Просмотр чужой ленты — 0.5 (read)
  - Места:
    - Просмотр постов по местам — 2 (read)
    - Поиск популярных мест — 0.2 (read)

Чтений на порядок больше, чем записей — система read-heavy.

Распределение пользователей (90-9-1): ~90% только читают, ~9% реагируют, ~1% создают контент. Блогеры (до 1M подписчиков) дают пики записи при публикации.

Пиковую (внутридневную) нагрузку считаем x3 от средней. Сезонный множитель учитываем отдельно.


- Сезонность
  - Выраженные сезоны: летние месяцы (отпуска), новогодние и майские праздники
  - Пиковый сезонный множитель x2 к средней нагрузке (применяется поверх внутридневного пика x3)

- Условия хранения данных
  - Контент (посты, фото, комментарии) храним бессрочно — история путешествий это ценность сервиса
  - При удалении профиля контент анонимизируется ("Удаленный пользователь"), физически может храниться ограниченный срок для бэкапов

- Лимиты и ограничения
  - Максимум подписок у пользователя — 7 500 (анти-спам)
  - Количество подписчиков — без жесткого лимита (у блогеров до ~1 000 000)
  - Максимум фото на пост — 10
  - Максимальный размер одного фото — 5 МБ
  - Максимальная длина описания поста — 2 000 символов
  - Максимальная длина комментария — 1 000 символов

- Временные ограничения (тайминги, p99)
  - Чтение (лента, посты, профиль) — < 1 с
  - Запись (пост, комментарий, оценка) — < 2 с
  - Система не относится к low-latency, агрессивные SLA не требуются

- Доступность приложения
  - 99.95% (~4.4 часа простоя в год)
  - Не 99.999%: система не критична к минутам простоя; 99.95% обеспечивает надежность под бизнес-требование при разумной стоимости

- Надежность данных (durability)
  - Потеря пользовательского контента (посты, фото, комментарии) недопустима

- Масштабируемость
  - Система должна выдерживать рост нагрузки до 20M DAU (горизонт 2 года) без редизайна архитектуры

- Клиенты
  - Мобильные устройства и браузер


## Оценка нагрузки (RPS)

- Базовый множитель: DAU / 86400 = 10 000 000 / 86400 ~ 115.7
- RPS(средний) ~ 115.7 x (действий в сутки на пользователя)
- RPS(пик) = RPS(средний) x 6 (внутридневной пик x3 x сезонный множитель x2)
- Размер страницы ленты/мест — 10 постов за запрос

### Профиль
- read (просмотр профиля): 115.7 x 3 ~ 347 | пик ~ 2 080
- write: редкие (edit/delete), пренебрегаем

### Посты
- write (публикация): 115.7 x 0.01 ~ 1.2 | пик ~ 7
- read (чтение постов): см. "Критический путь" ниже

### Комментарии
- write (создание): 115.7 x 0.05 ~ 5.8 | пик ~ 35
- read (чтение): 115.7 x 5 ~ 578 | пик ~ 3 470

### Оценки
- write (поставить/убрать): 115.7 x 0.5 ~ 58 | пик ~ 347

### Подписки
- write (подписка/отписка): 115.7 x 0.01 ~ 1.2 | пик ~ 7

### Лента
- read своя: 115.7 x 5 ~ 578 | пик ~ 3 470
- read чужая: 115.7 x 0.5 ~ 58 | пик ~ 347

### Места
- read (просмотр постов места): 115.7 x 2 ~ 231 | пик ~ 1 390
- read (поиск популярных мест): 115.7 x 0.2 ~ 23 | пик ~ 139

### Критический путь
По числу запросов (RPS) самые нагруженные — чтение ленты (~578) и чтение комментариев (~578). Запись на 2–3 порядка ниже (макс. оценки ~58 RPS). Система read-heavy.

### Размеры сущностей

Метаданные хранятся в БД, фото и аватары — в объектном хранилище/CDN (в БД только ссылка). Размеры текстов — средние значения (лимиты в НФТ — максимумы).

| Сущность | Метаданные | Медиа |
| --- | --- | --- |
| Пост | ~360 B | — (фото вынесено в Фото) |
| Фото | ~156 B | ~500 KB, в среднем 3 на пост |
| Комментарий | ~232 B | — |
| Лайк | 24 B | — |
| Подписка | 24 B | — |
| Профиль | ~336 B | аватар ~50 KB |
| Место | ~400 B | — |

- Лайк: user_id 8, post_id 8, created_at 8 -> ~ 24 B
- Подписка: follower_id 8, following_id 8, created_at 8 -> ~ 24 B

## Оценка нагрузки — трафик

- Traffic = RPS x размер ответа
- Медиа (фото, аватары) раздается через CDN — считаем отдельно от метаданных
- Размер поста с медиа = 360 B + 3 фото x 500 KB ~ 1.5 MB
- Egress (download/read) доминирует

### Лента (egress, read)
- своя (578 RPS), ответ = 10 постов x (360 B мета + 1.5 MB медиа):
  - метаданные ~ 578 x 10 x 360 B ~ 2.1 MB/s
  - медиа (CDN) ~ 578 x 10 x 1.5 MB ~ 8.7 GB/s
- чужая (58 RPS): метаданные ~ 0.2 MB/s, медиа ~ 0.9 GB/s

### Места (egress, read)
- просмотр постов (231 RPS x 10 постов): метаданные ~ 0.8 MB/s, медиа ~ 3.5 GB/s
- поиск (23 RPS x 10 мест x 400 B): ~ 0.1 MB/s

### Профиль (egress, read)
- 347 RPS x (336 B мета + 50 KB аватар): метаданные ~ 0.1 MB/s, медиа (CDN) ~ 17 MB/s

### Комментарии
- read (578 RPS x 10 комментов x 232 B): ~ 1.3 MB/s (метаданные)
- write (5.8 RPS x 232 B): ~ 1.3 KB/s

### Посты (ingress, write)
- публикация (1.2 RPS x (360 B мета + 1.5 MB upload)): метаданные ~ 0.4 KB/s, медиа ~ 1.8 MB/s

### Оценки / Подписки (ingress, write)
- оценки 58 RPS x 24 B ~ 1.4 KB/s; подписки 1.2 RPS x 24 B ~ 30 B/s

### Итого (средний трафик)
- Egress (download): медиа через CDN ~ 13 GB/s, метаданные ~ 5 MB/s
- Ingress (upload): медиа ~ 1.8 MB/s, метаданные ~ 3 KB/s
- Пик (x6): egress медиа ~ 78 GB/s

Вывод: трафик полностью доминируется медиа на чтение (раздача фото из ленты и мест через CDN); метаданные и запись малы.

## Оценка нагрузки — соединения

Одновременные соединения ~ доля пользователей, онлайн в один момент (~10% DAU):

Connections = DAU x 0.1 = 10 000 000 x 0.1 = 1 000 000

~1M одновременных соединений — влияет на число и тип балансировщиков, gateway и размер connection pool, особенно при long-lived соединениях для обновления ленты в реальном времени.

## Модель хранения данных

Данные разнесены по хранилищам по паттерну доступа. Схемы — в директории `database/` (по файлу на хранилище), фотографии — в объектном хранилище MinIO.

| Хранилище | Что хранит | Почему это хранилище |
|---|---|---|
| PostgreSQL | профили, посты, фото, комментарии, реакции, подписки | source of truth: транзакции и целостность |
| Elasticsearch | поисковые документы постов и мест | полнотекстовый поиск |
| Redis | read-model постов со счетчиками, ленты | cache-aside поверх PostgreSQL, горячее чтение |
| MinIO | фотографии | объектное хранилище блобов |

<details>
<summary>Ключевые решения</summary>

- Реакции. Факт лайка/дизлайка надежно хранится в `post_reactions` (PK `post_id + user_id`, одна реакция на пост от пользователя) — это истина. Счетчик денормализован в `posts.like_count` (восстановим `COUNT(*)`). В кэше счетчик обновляется по стратегии write-behind: на лайк `INCR` поля в `post_read_models` (мгновенно), `posts.like_count` флашится в PostgreSQL фоновым процессом пачкой (eventual consistency). Снапшот инвалидируется только при редактировании поста — на лайк счетчик правится через `INCR`, пересобирать весь снапшот не нужно.
- Фото. Принадлежат посту (1:N), `photos.post_id` nullable — фото грузится до создания поста (pre-upload) и привязывается при публикации.
- Комментарии. Привязаны к посту, ветки ответов — через ссылку на родительский комментарий (`parent_id`).
- Подписки. `UNIQUE (follower_id, following_id)` — идемпотентность.
- Локация. Пост хранит координаты (`latitude`/`longitude`) — где сделано фото, в том числе безымянная точка (трасса, поле). `place_id` опционален: ссылка на именованное место из справочника `places`, если оно есть. Гео-поиск по близости в текущих ФТ не требуется (поиск идет по местам-справочнику), поэтому храним простые координаты без spatial-типов.

</details>

### Допущения

- Удаление профиля — анонимизация: `display_name = 'Удаленный пользователь'`, `bio`/`avatar_url = NULL`, контент сохраняется, без soft-delete.
- Комментарии — ФТ требуют плоские комментарии, `parent_id` заложен как расширение под ветки ответов.
- Места — модерируемый справочник, при создании поста выбирается существующее место (оценка 1M).

### PostgreSQL

Объем — по средним размерам строк за 1 год (лимиты НФТ — максимумы). Коэффициент x2 — индексы и запас на рост.

#### Итоговый объем

| Сущность               | Расчет годового объема | Годовой объем | Коэффициент | Итоговый объем |
|------------------------|------------------------|--------------:|------------:|---------------:|
| Посты                  | 36 500 000 x 360 B     |      13.14 GB |          x2 |          26 GB |
| Комментарии            | 182 500 000 x 232 B    |      42.34 GB |          x2 |          85 GB |
| Лайки (post_reactions) | 1 825 000 000 x 24 B   |      43.80 GB |          x2 |          88 GB |
| Фото (метаданные)      | 109 500 000 x 156 B    |      17.08 GB |          x2 |          34 GB |
| Подписки               | 36 500 000 x 24 B      |       0.88 GB |          x2 |           2 GB |
| Профили (travelers)    | 10 000 000 x 336 B     |       3.36 GB |          x2 |           7 GB |
| Места (places)         | 1 000 000 x 400 B      |       0.40 GB |          x2 |           1 GB |
| Итого              |                        |   ~121 GB |             |    ~243 GB |

Число строк из раздела «Оценка нагрузки (RPS)» x секунды в году: лайки 58 RPS avg x 31.5M с ~ 1.825 млрд/год, подписки 1.2 RPS avg -> 36.5M/год, фото — 3 на пост x 36.5M постов = 109.5M.

#### Расчет количества дисков

Тип диска: SSD SATA — PostgreSQL это транзакционный write-model (source of truth), нужен random IOPS и запись с latency < 2 c.

Пиковые RPS записи — из раздела «Оценка нагрузки (RPS)» (пик = avg x 6):

| Параметр        | Расчет                                                    | Значение |
|-----------------|-----------------------------------------------------------|---------:|
| Peak Write IOPS | 7 (посты) + 35 (комментарии) + 347 (лайки) + 7 (подписки) + 21 (фото) |     ~417 |
| Peak Throughput | метаданные записи (медиа — в MinIO, не в PG)               | < 1 MB/s |
| Capacity        | объем данных за год                                        |  ~243 GB |

Расчет:

```text
Disks_for_capacity   = 243 GB / 1 TB = 0.243
Disks_for_throughput = <1 / 500      ~ 0.002
Disks_for_iops       = 417 / 1000    = 0.417

Disks = max(
    ceil(0.242),
    ceil(0.002),
    ceil(0.417)
) = 1
```

### Elasticsearch

Тип диска: SSD SATA — поиск latency-sensitive, много random-чтений по индексу.

Поисковые документы: постов и мест. Объем ~2x исходных данных из-за инвертированного индекса и служебной информации.

| Документ  | Расчет годового объема | Годовой объем | Коэффициент | Итоговый объем |
|-----------|------------------------|--------------:|------------:|---------------:|
| Посты     | 36 500 000 x 1 KB      |       36.5 GB |          x2 |          73 GB |
| Места     | оценочно               |          5 GB |          x2 |          10 GB |
| Итого |                        |   41.5 GB |             |      83 GB |

#### Расчет количества дисков

Пиковый поиск — из раздела «Оценка нагрузки (RPS)» (поиск популярных мест, пик ~139):

| Параметр         | Расчет               | Значение |
|------------------|----------------------|---------:|
| Peak Search IOPS | 139 RPS              |      139 |
| Peak Throughput  | 139 x 50 KB          |   7 MB/s |
| Capacity         | объем индекса за год |    83 GB |

```text
Disks_for_capacity   = 83 GB / 1 TB = 0.083
Disks_for_throughput = 7 / 500      = 0.014
Disks_for_iops       = 139 / 1000   = 0.139

Disks = max(
    ceil(0.083),
    ceil(0.014),
    ceil(0.139)
) = 1
```

### Redis

Redis — кэш горячих данных поверх PostgreSQL (source of truth), читаем из кэша, при промахе — из БД (cache-aside):

- read-model горячих постов со счетчиками (`post_read_models`)
- кэш лент пользователей (`traveler_feed_read_models`)
- кэш постов по местам (`place_posts_read_models`)

В RAM держим только горячее подмножество: малая доля постов и лент (свежие, популярные) собирает основную часть чтений, остальное перечитывается из PostgreSQL. Вытеснение редко используемого — политикой `allkeys-lru`.

---

#### 1. RAM (горячее подмножество)

Оцениваем горячий рабочий набор (ориентир — активность ~последних 30 дней), а не весь годовой корпус.

| Подсистема       | Расчет                    |          Объем |
|------------------|---------------------------|---------------:|
| Post read models | 3M горячих x 1 KB         |           3 GB |
| Traveler feeds   | 1M активных x 3 KB        |           3 GB |
| Place feeds      | оценочно                  |           1 GB |
| Подытог          |                           |         ~7 GB |
| Итого (+~30% на накладные расходы) |     | ~9 GB RAM |

Значение зависит от TTL и размера рабочего набора — при росте кэша `allkeys-lru` вытесняет холодное, потолок задается `maxmemory`.

---

#### 2. Persistence (диск)

Redis — кэш, данные восстановимы из PostgreSQL, поэтому persistence опциональна: без нее кэш прогревается заново, с RDB-снапшотом размер ~ датасету (~9 GB), а не сумма операций за год.

#### Итог

| Ресурс      |           Значение |
|-------------|-------------------:|
| RAM         |            ~9 GB   |
| Disk        | 0 (кэш) или ~9 GB (RDB) |
| Тип диска   | SSD SATA (если RDB) |

Отказоустойчивость Redis обеспечивается репликой (master + replica), а не дисковым запасом.

### Хранение фотографий

Для хранения фотографий используем MinIO.
Фотографии распределяются между storage-узлами по Consistent Hashing от `photo_id`, каждая фотография хранится в `3` копиях на разных хостах.

Тип диска: HDD — для фото важен объем, а не скорость доступа, горячее чтение снимает CDN, throughput диска с запасом перекрывает запись. SSD за ГБ дороже без выигрыша для этого профиля.

Размер приведен в соответствие с ДЗ 1: в среднем 3 фото на пост x ~500 KB = 1.5 MB/пост.

| Параметр                  |                              Значение |
|---------------------------|--------------------------------------:|
| Средний размер фотографии |                                500 KB |
| Фото на пост (среднее)    |                                     3 |
| Публикаций в сутки        |                               100 000 |
| Фото в год                |          100 000 x 3 x 365 = 109.5 млн |
| Logical объем в год       | 109.5 млн x 500 KB ~ 54.75 TB     |
| Фактор репликации         |                                  x3   |
| Raw объем (с репликацией) |          54.75 TB x 3 ~ 164.25 TB |

Расчет числа дисков (HDD 18 TB, целевое заполнение ~80% -> usable ~ 14.4 TB/диск):

```text
Disks_for_capacity = raw_объем / usable_на_диск = 164.25 TB / 14.4 TB = 11.4
Disks = ceil(11.4) = 12   (репликация x3 уже учтена в raw)
```

12 дисков распределяем по >=3 узлам (по 4 диска) — так три копии каждого фото
ложатся на разные хосты (consistent hashing по `photo_id`). Плюс hot-spare
на отказ и рост в течение года.

## Распределенное хранение данных

### Шардирование

PostgreSQL не шардируем. Объем данных за год мал и укладывается в один хост, а система read-heavy, поэтому шардирование для распределения записи не нужно. На горизонте 2 года (20M DAU) пересмотреть (триггеры: размер PG кратно больше RAM, рост write RPS, replication lag).

Elasticsearch и MinIO распределяют данные штатными механизмами: ES делит индекс на primary и replica shards, MinIO раскладывает объекты по узлам через consistent hashing (каждое фото в 3 копиях на разных узлах).

### Масштабирование на горизонте 2 года

- PostgreSQL: шардируем по user_id (hash), добавляем read-реплики под чтение ленты и профиля.
- MinIO: расширяем кластер добавлением узлов, новые объекты идут на узлы с большим свободным местом, ребаланс существующих запускаем отдельно при необходимости.
- Redis: переходим на Redis Cluster (hash slots), кэш растет горизонтально.
- Elasticsearch: добавляем data-узлы, при необходимости пере-шардируем индекс в новый с большим числом shards.

### Репликация

| Хранилище | Тип | RF | Зачем |
|-----------|-----|---:|-------|
| PostgreSQL | master-slave, async | 3 | отказоустойчивость и разгрузка чтения с реплик (read-heavy) |
| Elasticsearch | primary-replica | 2 | отказоустойчивость индекса |
| Redis | Sentinel (master-slave) | 2 | failover кэша, данные восстановимы из PostgreSQL |
| MinIO | 3 копии на разных хостах | 3 | durability фото (учтено в дисках) |

RF 3 = 3 копии (1 master + 2 replica), RF 2 = 2 копии (master + 1 replica). async для PostgreSQL: соцсеть не критична к потере последних секунд, реплики разгружают чтение. Чтение с async-реплик может отставать от мастера (replication lag) — для соцсети приемлемо. Redis RF 2: кэш, реплика нужна для доступности, не для durability.

Размещение: single-region, реплики на хостах, отличных от мастера (отказ одного хоста не теряет данные). Redis Sentinel — минимум 3 инстанса для кворума, отдельно от data-узлов.

### Оценка хостов на 1 год

Оцениваем по capacity (диски) — она доминирует при нашем read-heavy профиле (RPS малы, CPU/IOPS/сеть не узкое место). На проде хост считают по max(capacity, CPU, RAM, network, IOPS). От числа дисков: для PostgreSQL, Elasticsearch и Redis disks_per_host = 1, для MinIO берем 4 диска на хост (см. пояснение под таблицей). Формула: Hosts_with_replication = hosts x RF.

Подсистемы (посты, реакции, комментарии) живут в одной PostgreSQL и делят ее хосты, поэтому разрез по подсистемам сделан на уровне объема данных, а хосты считаем по хранилищу.

Важно: диски MinIO уже посчитаны с репликацией (raw = logical x 3), поэтому повторно на RF не умножаем.

| Подсистема | Диски | Хосты | RF | Хосты с репликацией |
|------------|-------|------:|---:|--------------------:|
| PostgreSQL | 1 SSD (~243 GB) | 1 | 3 | 3 |
| Elasticsearch | 1 SSD (~83 GB) | 1 | 2 | 2 |
| Redis | RAM ~9 GB | 1 | 2 | 2 |
| MinIO | 12 HDD (164 TB raw, уже x3) | 3 | — | 3 |
| Итого | | | | 10 |

MinIO: 12 дисков раскладываем по 4 на хост -> 3 хоста. Три копии каждого объекта ложатся на три разных хоста, это минимально корректная схема для RF 3 (потеря одного хоста не теряет данные).

### Охлаждение данных

Контент хранится бессрочно, но старые данные читаются редко, поэтому держать весь архив на дорогом SSD расточительно. Охлаждаем по возрасту:

- PostgreSQL: posts и comments партиционируем по created_at. Свежие партиции на SSD, старые на HDD, перенос по расписанию. Нюанс: партиционирование ускоряет запросы по дате (лента), но не выборки по post_id (комментарии поста).
- MinIO: старые фото переносим на более холодный tier (lifecycle transition) — экономия места, не замена бэкапу.

### Резервное копирование

Репликация защищает от отказа узла, но не от порчи или удаления данных (ошибка размножится на реплики), поэтому бэкапы держим отдельно:

- PostgreSQL: PITR (base backup + WAL-архив).
- Elasticsearch: snapshots (индекс восстановим reindex из PostgreSQL, но snapshot нужен для восстановления настроек индекса, не только данных).
- MinIO: версионирование или отдельный бэкап-бакет.

RPO/RTO: async-репликация допускает потерю последних секунд при отказе мастера (RPO > 0) — для соцсети приемлемо. Failover PostgreSQL автоматический, клиенты переключаются на новый мастер.
